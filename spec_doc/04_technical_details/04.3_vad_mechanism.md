# VAD 与分段策略

## 概述

系统使用 ASR 上游事件来驱动候选人分段，统一通过适配器归一化为内部标准事件，再交给 `RealtimeSessionRunner` 处理。

内部标准事件：
- `speech_started`
- `speech_stopped`
- `transcript_completed`
- `input_committed`
- `error`

代码位置：
- `backend/app/services/realtime/asr_adapter.py`
- `backend/app/services/realtime/session_runner.py`
- `backend/app/services/realtime/audio_pipeline.py`

## 事件流

1. 前端持续发送 `audio` 给后端
2. 后端按 `ARK_ASR_MODE` 连接对应 ASR 上游
3. 上游事件经 `AsrAdapter.normalize_event(...)` 转换为标准事件
4. `speech_started` 时开启分段缓存
5. `speech_stopped` 时封段并触发 `finalize_candidate_segment(...)`
6. `transcript_completed` 到达后进入决策与回复生成

## 降级策略

某些 ASR 模式可能不稳定提供 `speech_*` 事件。当前实现支持降级：
- 依赖前端显式 `end_turn`
- 或由 `no_response_timeout` 驱动提交

这样可保证流程不死锁，即便自动 VAD 行为降级，面试仍可继续。

## 为什么仍然持续上送音频

即使未进入“候选人有效分段”，后端仍会将音频转发到 ASR 上游（由具体模式决定），原因是：
- 上游检测边界需要连续上下文
- 保持实时连接稳定
- 让 `speech_started` 更及时

## 与回合编排关系

VAD 只负责“分段边界”与“转写可用时机”，不负责业务推进。业务推进必须满足：
1. 分段完成
2. 转写可用
3. 决策完成
4. 本轮 `response.done` 完成

只有最后一步完成，状态机才推进到下一业务阶段。
