# 100 并发面试承载优化清单

本文档面向“**同时 100 场实时语音面试**（100 并发 WebSocket 会话 + 100 条 OpenAI Realtime 上游连接）”这一目标，对当前架构提出优化清单。每条优化都包含：

- **解决什么问题**：当前瓶颈/风险点
- **怎么做**：建议实现方式（可落地）
- **优点**：带来的收益
- **缺点/麻烦点**：代价、复杂度、潜在副作用
- **验收/验证**：如何证明它确实提升了 100 并发承载能力

> 术语说明  
> - **下游**：浏览器 ↔ 后端（`/api/realtime/ws/{token}`）  
> - **上游**：后端 ↔ OpenAI Realtime（`wss://api.openai.com/v1/realtime?...`）  
> - **会话**：一场面试的端到端长连接与状态（通常持续 10–30 分钟）

---

## 目标与非目标

### 目标
- **稳定承载 100 并发面试会话**：低错误率、可恢复、可观测
- **在高并发下保持交互体验**：VAD/转发延迟不出现明显抖动（卡顿、断流）
- **在外部限流下可控降级**：遇到配额/429 时优雅失败或排队

### 非目标（本清单不覆盖或仅建议）
- “无限并发”或跨区域多活
- 复杂的计费系统、全链路审计体系
- 音频内容的深度安全合规（需要单独文档）

---

## 0. 先做的事实核对（不改代码也能立刻降低不确定性）

### 0.1 确认 OpenAI Realtime 的并发与速率上限
- **解决什么问题**
  - 100 并发最常见的第一瓶颈不是你服务端，而是 **上游并发连接数/吞吐/速率限制**。
- **怎么做**
  - 向 OpenAI 控制台/支持渠道确认当前项目对 Realtime 的：
    - 同时连接数上限（并发 session）
    - 每分钟音频/请求/令牌限制
    - 429/限流策略与重试建议
  - 在压测环境中模拟 100 条上游连接，观察 429/断连模式。
- **优点**
  - 直接决定“是否可能”，避免在本地工程上做无效优化。
- **缺点/麻烦点**
  - 需要外部配合；额度可能随时间变化。
- **验收/验证**
  - 100 条上游连接可建立并稳定保持 N 分钟（如 15 分钟）且错误率在可接受范围内。

### 0.2 明确“100 并发”的真实含义
- **解决什么问题**
  - “100 人同时面试”可能是：100 条长连接、100 人同时说话、或 100 场同时在“AI 说话/人说话”的不同阶段。不同假设导致带宽、CPU、外部费用完全不同。
- **怎么做**
  - 把目标固化成 2–3 个场景（例如：平均 50% 时间用户说话、峰值 20 人同时说话，等）。
- **优点**
  - 让压测与容量评估可复现、可比较。
- **缺点/麻烦点**
  - 需要业务侧给出合理分布假设。
- **验收/验证**
  - 压测脚本/数据能覆盖这 2–3 个明确场景，并输出一致指标。

## 1. 必须项（不做基本无法稳定 100 并发）

### 1.1 生产化部署：多 worker / 多实例 + 反向代理正确配置
- **解决什么问题**
  - 单进程/单事件循环在 100 条长连接下容错差；任何阻塞都会放大抖动。
  - 需要正确的 WebSocket 反向代理配置，避免 upgrade/keepalive 问题。
- **怎么做**
  - 使用 `gunicorn -k uvicorn.workers.UvicornWorker -w <N>` 或容器多副本部署。
  - 前置 Nginx/Envoy/ALB，开启 WebSocket 支持、合理的超时（读超时、空闲超时）。
  - **按 CPU 核数估算 worker**：先从 2–4 起步压测，再调。
- **优点**
  - 横向扩展能力立刻出现；单 worker 崩溃不致全体断线（配合多实例）。
- **缺点/麻烦点**
  - 会话粘性/负载均衡需要考虑（WebSocket 必须粘到同一实例直到断开）。
  - 调试复杂度上升。
- **验收/验证**
  - 任一实例重启不会影响其它实例会话；100 并发连接在多实例下分布均衡、无大量 101/upgrade 错误。

### 1.2 数据库从 SQLite 切换到 PostgreSQL（并启用连接池）
- **解决什么问题**
  - SQLite 并发写锁会导致抖动甚至请求堆积；并发越高越明显。
- **怎么做**
  - 生产使用 PostgreSQL；SQLAlchemy 配置连接池（pool_size、max_overflow）。
  - 为关键查询字段建索引（如 `Interview.link_token`）。
- **优点**
  - 并发写入能力显著提升，降低“偶发卡死/锁等待”。
- **缺点/麻烦点**
  - 需要运维数据库、备份、迁移策略。
- **验收/验证**
  - 100 并发压测下 DB 事务耗时分布稳定（p95/p99 无离谱尖峰），无明显锁等待告警。

### 1.3 把“同步阻塞 IO”移出 WebSocket 事件循环（音频落盘 + DB commit）
- **解决什么问题**
  - 当前在 VAD `speech_stopped` 里进行 WAV 封装、文件写入、`db.commit()` 属于同步阻塞操作；并发时会阻塞该 worker 的事件循环，造成所有连接抖动。
- **怎么做**
  - 方案 A（推荐）：引入队列（Redis + RQ/Celery/Arq），把“保存音频文件/写 Answer/后处理”异步化。
  - 方案 B（简化）：使用 `asyncio.to_thread(...)` 将落盘与 DB 写入放到线程池（仍建议最终过渡到队列）。
  - 方案 C：直接存对象存储（S3/OSS）+ 异步上传（仍建议后台任务）。
- **优点**
  - WebSocket 转发路径“变薄”，延迟更稳定；更容易扩大并发。
- **缺点/麻烦点**
  - 引入任务队列与一致性问题（写入延迟、失败重试、幂等）。
  - 需要补齐任务追踪与死信/重试策略。
- **验收/验证**
  - 在压测下，WebSocket 消息循环延迟（event loop lag）显著下降；连接的音频转发无周期性卡顿。

### 1.4 上游限流与保护：并发连接、消息频率、异常风暴
- **解决什么问题**
  - 没有限流时，异常（比如客户端重连风暴、上游 429）会把后端与上游同时打爆。
- **怎么做**
  - **全局并发面试数**上限（例如 100/120）+ 超出返回明确错误/排队。
  - **每 token 连接互斥**：同一 `link_token` 只允许 1 个活跃会话（防止重复打开造成资源翻倍）。
  - **音频 append 频率保护**：限制每秒 append 次数/字节数，超出则丢弃或降采样（客户端优先）。
  - **上游连接失败退避**：指数退避 + 抖动，避免瞬时雪崩。
- **优点**
  - 把“不可控的并发”变成“可控的拒绝/排队/降级”。
- **缺点/麻烦点**
  - 需要定义业务策略（排队、提示文案、重试窗口）。
  - 做不好可能影响真实用户体验（被误限流）。
- **验收/验证**
  - 人为制造异常风暴（断网/重连/429）时，后端仍然可用且不会全局崩溃；错误可解释、可恢复。

## 2. 高优先级（强烈建议做：显著提升稳定性/成本/排障能力）

### 2.1 连接生命周期治理（断线、重连、半开连接清理）
- **解决什么问题**
  - WebSocket 场景常见“半开连接”或异常断线未清理，导致资源泄露（上游连接仍占用、内存增长）。
- **怎么做**
  - 设置心跳（ping/pong）与超时：下游、上游都要有。
  - 明确 `finally` 清理：确保上游 ws 关闭、释放缓冲区。
  - 对“浏览器掉线但上游还连着”的情况，强制关闭上游。
- **优点**
  - 降低长时间运行后的内存/连接泄漏风险。
- **缺点/麻烦点**
  - 需要仔细处理各种异常路径，测试工作多。
- **验收/验证**
  - 长时间压测（例如 2–4 小时）连接数与内存曲线平稳，无持续上升。

### 2.2 可观测性：把“能不能扛住 100 并发”变成可量化指标
- **解决什么问题**
  - 没有指标就只能靠感觉，定位抖动来源（DB？磁盘？上游？网络？）会非常慢。
- **怎么做**
  - 关键指标（建议至少有）：
    - 当前活跃会话数（下游、上游分别统计）
    - event loop lag / worker CPU / 内存
    - VAD 段保存耗时（p50/p95/p99）
    - DB commit 耗时、连接池耗尽率
    - 上游错误码分布（429/5xx/断连）
    - 每会话音频上行/下行字节与速率
  - 导出到 Prometheus/Grafana 或至少结构化日志 + 仪表盘。
- **优点**
  - 容量评估、回归验证、故障定位速度大幅提升。
- **缺点/麻烦点**
  - 需要搭监控栈；指标埋点需要统一规范。
- **验收/验证**
  - 能回答：100 并发时 CPU/内存/DB/上游分别占多少？瓶颈在哪？提升后是否变好？

### 2.2.1 决策层（独立 LLM）延迟预算与降级策略
- **解决什么问题**
  - 新增独立决策调用后，如果没有超时预算与兜底，可能把每轮停顿放大，导致用户体感“AI 反应慢”。
- **怎么做**
  - 为决策调用设置严格超时（建议 600-900ms）。
  - 决策失败（超时、解析失败、API 异常）立即回退 legacy rule planner。
  - 决策上下文只保留最小必要信息（当前阶段、当前题、最新发言、最近 2-4 轮摘要）。
  - 监控 `decision_layer.fallback` 比例与 `decision_layer.succeeded.latency_ms` 分位数。
- **优点**
  - 在提升流程语义判断能力的同时，把交互延迟控制在可接受范围。
- **缺点/麻烦点**
  - 增加一次文本 LLM 成本；需要维护 action 映射与回退逻辑。
- **验收/验证**
  - 100 并发场景下，决策层 p95 延迟在预算内（例如 <= 900ms），且 fallback 后面试可持续推进不中断。

### 2.2.2 单一线性链路对稳定性的收益
- **解决什么问题**
  - `speech_stopped`、`end_turn`、转写晚到、重复 `response.done` 等事件同时存在时，容易出现双触发和状态错位。
- **怎么做**
  - 将实时会话统一为 `candidate_audio -> transcription -> decision -> ai_response`。
  - 使用 `pipeline_lock`、`decision_pending`、`pending_plan`、`pipeline_stage` 保证一次只推进一个 segment。
  - 只在 `response.done(completed)` 后应用 `BusinessTransition`。
- **优点**
  - 大幅减少误跳题、误 closing、重复推进等竞态问题。
  - 更容易做压测、复盘和结构化观测。
- **缺点/麻烦点**
  - 需要维护更严格的状态机和更系统的测试。
- **验收/验证**
  - 并发 finalize、晚到 transcript、重复 done 等场景下仍能保持正确顺序和幂等。

### 2.3 音频与消息开销：减少 base64/JSON 频次带来的 CPU 与带宽浪费
- **解决什么问题**
  - base64 + JSON 封包在高并发下会增加 CPU 与带宽；100 并发可能把 CPU 花在“编码/解析”而非业务上。
- **怎么做**
  - 优先：调大客户端音频 chunk（在可接受延迟内）降低消息次数。
  - 进阶：采用二进制 WebSocket 帧（避免 base64），或使用更高效的封包格式。
  - 评估是否要把“音频转发网关”与“业务 API”拆服务（避免互相影响）。
- **优点**
  - 减少 CPU 与 GC 压力；更平滑的延迟曲线。
- **缺点/麻烦点**
  - 二进制帧/协议变更会影响前后端实现与兼容性。
- **验收/验证**
  - 在同等并发下，CPU 降低、吞吐提升、端到端延迟更稳定。

### 2.4 会话状态外置（多实例下的粘性与恢复）
- **解决什么问题**
  - 多实例下如果实例重启，内存态会丢；需要更好的恢复与状态一致性。
- **怎么做**
  - 把“会话元数据/进度”写入 Redis（轻量）或 DB（更强一致）：
    - 当前 stage、题号、followup 次数、是否 overtime
  - 让新实例能在必要时恢复/终止旧会话（通常 WebSocket 不可无缝迁移，但至少可“可控结束/提示重连”）。
- **优点**
  - 运维更安全；故障影响更可控。
- **缺点/麻烦点**
  - 需要设计状态机与幂等写入；实现成本中等。
- **验收/验证**
  - 滚动发布/单实例重启时，系统对用户的表现可预测（明确提示、可重连、不会出现幽灵会话占资源）。

## 3. 可选项（视目标、成本、复杂度选做）

### 3.1 评估（`/complete`）链路异步化与排队
- **解决什么问题**
  - 评估通常是 LLM +（可能的）STT，属于重任务；高并发下容易把 API 打满，影响实时面试网关。
- **怎么做**
  - 把评估改为后台任务：立即返回“已提交，评估中”，前端轮询/订阅评估完成事件。
  - 评估任务设置并发度与队列优先级（避免挤占实时会话资源）。
- **优点**
  - 实时链路与重计算解耦，提升整体稳定性。
- **缺点/麻烦点**
  - 需要任务状态、重试、前端展示“评估中”。
- **验收/验证**
  - 评估高峰时实时面试仍稳定；评估平均完成时间可预测。

### 3.2 音频存储与回放的生产化（对象存储 + CDN）
- **解决什么问题**
  - 本地磁盘容量、IO、备份、跨实例读取都会变复杂；100 并发会产生大量音频文件。
- **怎么做**
  - 使用 S3/OSS 存储音频；回放走 CDN；DB 存 URL 与元数据。
- **优点**
  - 降低后端磁盘压力；跨实例一致；扩容更容易。
- **缺点/麻烦点**
  - 引入存储成本与权限管理；需要签名 URL/访问控制。
- **验收/验证**
  - 多实例部署下任意实例都能访问历史音频；音频读写成功率高。

### 3.3 成本优化：减少重复 STT / 使用对话日志替代转写
- **解决什么问题**
  - 当前如果“实时转写 + 完成后再 STT”会重复花钱；并发上来成本会很快失控。
- **怎么做**
  - 如果 Realtime 已提供可靠 transcript：评估直接用对话 transcript，不再对 WAV 走二次 STT。
  - 若 transcript 不可靠：只对缺失片段补 STT。
- **优点**
  - 显著降低成本，减少评估链路耗时。
- **缺点/麻烦点**
  - 需要评估“实时 transcript 的质量与完整性”；可能要做对齐/纠错。
- **验收/验证**
  - 在同等业务量下，STT 调用次数下降；评估质量不显著下降。

### 3.4 拆分服务：Realtime 网关与 Admin/CRUD 分离
- **解决什么问题**
  - 实时音频网关对延迟敏感，而管理后台/CRUD 对延迟不敏感但可能有慢查询；混在同一服务会互相拖累。
- **怎么做**
  - 拆成 `realtime-gateway`（只做 WS + 转发）与 `api-service`（CRUD、评估、导出）。
- **优点**
  - 隔离资源与故障域；扩容更精准。
- **缺点/麻烦点**
  - 部署与跨服务调用复杂度上升（鉴权、追踪、网络策略）。
- **验收/验证**
  - Admin 慢查询/导出压力不会影响实时链路指标（延迟、断线率）。

## 4. 推荐落地顺序（最短路径到 100 并发）

1. **核对上游配额**（0.1）+ 定义并发场景（0.2）
2. **生产化部署**（1.1）+ **PostgreSQL**（1.2）
3. **把落盘/DB commit 移出事件循环**（1.3）
4. **加并发/速率保护与重试退避**（1.4）
5. **补齐连接治理与可观测性**（2.1、2.2）
6. 再做性能细化与成本优化（2.3、3.x）

## 5. 压测与验收建议（面向 100 并发）

### 5.1 必测指标（建议写进压测报告）
- **稳定性**
  - 连接成功率、断线率（下游/上游）
  - 429/5xx 分布
- **体验**
  - 音频下行卡顿率（可用“连续 delta 间隔”统计）
  - 端到端延迟（从用户发声到 AI 开始回复的分布）
- **资源**
  - CPU、内存、event loop lag
  - DB 事务耗时与连接池耗尽率
  - 网络带宽（入/出）

### 5.2 验收门槛（示例，可按业务调整）
- 100 并发持续 15 分钟：
  - 断线率 < 1%
  - 429 可控（可重试或可解释拒绝），不出现全局雪崩
  - p95 延迟不超过目标阈值（由业务定义）

## 6. 当前实现进展（2026-03-13）

以下是针对“先稳住 20 并发”并为迈向 100 并发打基础的已落地项。

### 6.1 已落地
- **1.3 移出阻塞 IO（已实现）**
  - `input_audio_buffer.speech_stopped` 中的 WAV 封装、文件写入、`Answer` 入库已改为 `asyncio.to_thread(...)` 异步下沉执行。
  - 线程内使用独立 `SessionLocal()`，避免跨线程复用请求会话。
- **1.4 token 连接互斥（已实现）**
  - 同一 `link_token` 只允许一个活跃 WebSocket，会拒绝重复连接。
  - 在 `finally` 中回收活跃 token，避免“幽灵占位”。
- **单一线性链路重构（已实现）**
