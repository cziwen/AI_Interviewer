# 4.0 问题排查指南

## 启动前准备

### 环境要求
- **Python**: 3.9+
- **Node.js**: 16+
- **操作系统**: Linux/Mac（推荐）、Windows（需 WSL）

### 依赖安装

#### 后端依赖
```bash
cd backend

# 安装基础依赖
pip install -r requirements.txt

# 本地 STT/TTS 模式额外依赖
pip install RealtimeSTT RealtimeTTS

# 首次运行会自动下载 Whisper 模型（~150MB）
```

#### 前端依赖
```bash
cd frontend
npm install
```

### 启动流程

#### 1. 配置环境变量
```bash
# backend/.env
OPENAI_API_KEY=sk-your-openai-key
USE_LEGACY_OPENAI_REALTIME=false  # 本地模式
REALTIME_TTS_ENGINE=kokoro        # 可选: kokoro/coqui/orpheus
DATABASE_URL=sqlite:///./ai_interview.db
```

#### 2. 启动后端
```bash
cd backend

# 开发模式（自动重载）
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000

# 生产模式
uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 4
```

**首次启动预期**：
- Whisper 模型下载（约 1-2 分钟）
- TTS 引擎初始化（Kokoro 预热需 5-10 秒）
- 看到 `Application startup complete` 即成功

#### 3. 启动前端
```bash
cd frontend

# 开发模式
npm run dev

# 生产构建
npm run build
npm run preview
```

#### 4. 验证服务
```bash
# 检查后端健康
curl http://localhost:8000/docs

# 检查前端
open http://localhost:5173
```

### 模式切换

#### 切换到 Legacy OpenAI Realtime 模式
```bash
# 修改 .env
USE_LEGACY_OPENAI_REALTIME=true

# 重启后端（Ctrl+C 停止，然后重新运行 uvicorn）
```

**注意**：
- ⚠️ 切换模式后**必须重启后端**
- ✅ Legacy 模式不需要安装 RealtimeSTT/TTS
- ⚠️ Legacy 模式需要更多 OpenAI API 配额
- ✅ 本地模式首次启动会下载模型，耐心等待

## 常见问题与解决方案

### 1. WebSocket 连接问题

#### 1.1 连接无法建立
**症状**：前端显示"连接失败"或一直停留在"连接中"状态。

**可能原因**：
- 后端服务未启动
- Token 无效或已过期
- CORS 配置错误

**排查步骤**：
```bash
# 1. 检查后端是否运行
curl http://localhost:8000/health

# 2. 检查 token 有效性
curl http://localhost:8000/api/interviews/{token}

# 3. 查看后端日志
tail -f backend/logs/app.log | grep "WebSocket"
```

**解决方案**：
- 确保后端服务正常运行（`uvicorn app.main:app --reload`）
- 验证 token 未被使用或过期
- 检查 FastAPI CORS 中间件配置

#### 1.2 连接频繁断开
**症状**：面试进行中突然断连，需要刷新页面。

**可能原因**：
- 网络不稳定
- 后端处理异常未捕获
- 前端音频处理崩溃

**排查步骤**：
```javascript
// 前端添加详细错误日志
ws.onerror = (error) => {
  console.error('WebSocket error:', error);
};

ws.onclose = (event) => {
  console.log('WebSocket closed:', event.code, event.reason);
};
```

**解决方案**：
- 实现断线重连机制
- 后端添加全局异常处理器
- 前端捕获音频处理异常

### 2. 音频问题

#### 2.1 麦克风无声音
**症状**：音量指示器无波动，AI 无法接收到音频。

**可能原因**：
- 浏览器未授权麦克风权限
- 选择了错误的输入设备
- 麦克风硬件故障

**排查步骤**：
```javascript
// 检查麦克风权限
navigator.permissions.query({ name: 'microphone' }).then(result => {
  console.log('Microphone permission:', result.state);
});

// 检查音量门限
processor.onaudioprocess = (e) => {
  const rms = calculateRMS(e.inputBuffer.getChannelData(0));
  console.log('RMS:', rms, 'Threshold: 0.03');
};
```

**解决方案**：
- 在浏览器设置中允许麦克风权限
- 设备选择界面重新选择正确的麦克风
- 测试麦克风在其他应用中是否正常

#### 2.2 AI 语音无法播放
**症状**：可以看到 AI 文字回复，但听不到声音。

**可能原因**：
- 扬声器音量静音
- AudioContext 未正确初始化
- 音频解码失败

**排查步骤**：
```javascript
// 检查 AudioContext 状态
console.log('AudioContext state:', audioContext.state);

// 检查音频块接收
ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  if (data.type === 'audio_delta') {
    console.log('Received audio chunk, length:', data.audio.length);
  }
};
```

**解决方案**：
- 检查系统音量和扬声器设置
- 用户交互后再创建 AudioContext（浏览器策略要求）
- 验证 base64 解码逻辑正确性

#### 2.3 音频播放卡顿或延迟
**症状**：AI 声音断断续续或明显延迟。

**可能原因**：
- TTS 合成速度慢
- 网络带宽不足
- 播放队列逻辑错误

**排查步骤**：
```python
# 后端日志查看 TTS 延迟
logger.info(f"TTS TTFA: {ttfa:.2f}s")

# 检查音频队列堆积
print(f"Queue size: {audio_processor.audio_chunks.qsize()}")
```

**解决方案**：
- 更换更快的 TTS 引擎（Kokoro 优于 Coqui）
- 增加网络带宽或减小音频块大小
- 检查 `nextStartTimeRef` 计算逻辑

### 3. 转录问题

#### 3.1 转录完全无输出
**症状**：说话后没有任何文字反馈。

**可能原因**：
- RealtimeSTT 初始化失败
- Whisper 模型未下载
- 音量门限设置过高

**排查步骤**：
```python
# 检查转录器状态
logger.info(f"Recorder created: {self.recorder is not None}")

# 检查回调触发
def on_partial(text):
    logger.info(f"Partial: {text}")
```

**解决方案**：
```bash
# 手动下载 Whisper 模型
python -c "import whisper; whisper.load_model('base.en')"

# 降低音量门限测试
# frontend: rms > 0.01
```

#### 3.2 转录内容不准确
**症状**：转录文字与实际说话内容不符。

**可能原因**：
- 语言模型选择错误
- 背景噪音干扰
- 说话不清晰

**解决方案**：
- 确认语言设置正确（中文 `zh`，英文 `en`）
- 使用更好的麦克风或安静环境
- 升级 Whisper 模型（`base.en` → `medium` → `large`）

### 4. LLM 调用问题

#### 4.1 API 超时或失败
**症状**：AI 长时间无响应或返回错误信息。

**可能原因**：
- OpenAI API Key 无效或额度不足
- 网络连接到 OpenAI 失败
- Prompt 过长触发限制

**排查步骤**：
```bash
# 测试 API Key
curl https://api.openai.com/v1/models \
  -H "Authorization: Bearer $OPENAI_API_KEY"

# 检查后端日志
grep "LLM Error" backend/logs/app.log
```

**解决方案**：
- 验证 `.env` 中 `OPENAI_API_KEY` 正确性
- 检查 OpenAI 账户余额
- 使用代理或更换网络环境
- 简化 prompt 或减少上下文

#### 4.2 回复内容质量差
**症状**：AI 回答不自然或偏离主题。

**可能原因**：
- Prompt 设计不当
- 模型选择不合适
- 上下文信息不足

**解决方案**：
```python
# 优化 system prompt
system_prompt = """
你是一个专业且友好的面试官。
- 简洁：回复不超过 30 字
- 自然：使用过渡语言而非生硬的"下一题"
- 鼓励：适当肯定候选人的回答
"""

# 切换到更强模型
model="gpt-4o"  # 而非 gpt-3.5-turbo
```

### 5. 数据库问题

#### 5.1 答案未保存
**症状**：面试结束后查询不到回答记录。

**可能原因**：
- `db.commit()` 未执行
- 事务回滚
- 初始问候被错误保存

**排查步骤**：
```python
# 在 /voice_turn 中添加日志
logger.info(f"Saving answer: q_idx={question_index}, text={text[:50]}")
db.commit()
logger.info("Commit successful")
```

**解决方案**：
- 确保每次保存后调用 `db.commit()`
- 检查异常处理是否吞掉了错误
- 验证初始问候逻辑（`text != "你好"`）

#### 5.2 评分结果为空
**症状**：`evaluation_result` 字段为 `null`。

**可能原因**：
- `/complete` 未被调用
- GPT-4o 返回非 JSON 格式
- 评分逻辑异常

**排查步骤**：
```python
# 检查 /complete 调用
logger.info("Complete endpoint called")

# 验证 GPT-4o 响应格式
logger.info(f"Raw evaluation: {response.choices[0].message.content}")
```

**解决方案**：
- 前端确保点击"结束面试"后调用 `completeInterview(token)`
- 添加 JSON 解析异常处理
- 使用 `response_format={"type": "json_object"}` 强制 JSON 输出

### 6. 性能优化

#### 6.1 首次回复延迟高
**症状**：第一次 AI 回复需要等待 5-10 秒。

**原因**：
- TTS 引擎冷启动
- Whisper 模型首次加载

**解决方案**：
```python
# 预热 TTS 引擎
self.stream.feed("prewarm")
self.stream.play(muted=True)
```

#### 6.2 内存占用持续增长
**症状**：长时间运行后服务器内存不足。

**原因**：
- 音频队列未清理
- 转录缓存未释放
- WebSocket 连接泄漏

**解决方案**：
```python
# 定期清理队列
finally:
    audio_processor.audio_chunks = Queue()  # 重置队列
    transcription_processor.shutdown()
    ws.close()
```

```bash
# 恢复
cp backend/ai_interview.db.backup backend/ai_interview.db
```

### 重置面试状态
```sql
-- 将卡在 IN_PROGRESS 的面试重置
UPDATE interviews
SET status = 'CREATED'
WHERE status = 'IN_PROGRESS'
AND created_at < datetime('now', '-1 hour');
```
